# Security Model

What you trust, what you don't, and the risks stated honestly.

## You hold your own coins

- **Keys never leave your machine.** The engine (`pactd`) holds the BIP39 seed, derives every swap key from it, signs locally, and broadcasts its own transactions. Satchel persists no seed or passphrase, and the engine's RPC is loopback-only.
- **Refunds are automatic.** Every locked output has a timelock refund path. If a counterparty walks away, the engine refunds you once the timelock matures — it watches the chain and acts without you. The taker's deadline is always earlier than the maker's, so the maker cannot be stranded.
- **The seed is never stored in plaintext.** Choose a passphrase and the seed is encrypted with scrypt + ChaCha20-Poly1305 and stays *locked* until you unlock it in the engine's memory, per session. With no passphrase it is still wrapped at rest — on Windows and macOS under a key held in the OS keystore (Credential Manager / Keychain) that auto-unlocks so trading survives a reboot; on Linux it is only obfuscated and treated as unencrypted. The machine key protects the file at rest, not against someone already logged in as you — a passphrase is what defends against a local attacker. If that keystore key ever becomes unavailable (data dir copied to another machine, keychain reset), the engine detects it at startup and reports `needs_reimport` instead of failing lazily — Satchel opens a guided recovery-phrase re-import that re-provisions the wrap under a fresh machine key. Funds and swaps are unaffected; the machine's derive scope rotates (see multi-machine below), so the machine's own old in-flight swaps reappear as another machine's until taken over.

## The transports are untrusted and blind

- **They see ciphertext, not deals.** Coordination messages are sealed to the recipient (`PACTSEALED1`; on Nostr, gift-wrapped under a one-time key). An operator sees only a recipient hint and an opaque blob. **There is no plaintext downgrade** — the engine refuses any message that isn't sealed.
- **Offers are public on purpose.** An offer is a signed advert of terms; that's meant to be readable. Only the coordination that follows is sealed.
- **A relay can withhold, not steal.** The worst a board or relay can do is delay, drop, or censor messages — *liveness*, not *safety*. Funds stay protected by the timelocks. Mitigate withholding by posting to multiple boards and refreshing offers.

## No reputation, no fees, no middleman

- **Trust = atomicity only.** There are no scores, receipts, or reputation systems — the protocol's safety is the only thing you rely on. Bearer *slips* (private offers) are takeable by whoever holds them; safety comes from fixed terms, maker-funds-first, atomic settlement, and TTL expiry.
- **No platform fees.** `platform_fee_sat` is hard-wired to **0**. You pay only on-chain mining fees.

## Things worth knowing

- **You are your own custodian.** No one can recover your funds for you. Keep your recovery phrase backed up offline, and never share it — the whole model rests on you holding your keys.
- **Satchel won't adopt the wrong engine.** Before reusing a `pactd` already listening on the configured port, Satchel reads its cookie, calls `getinfo`, and verifies the network matches. If another network's instance holds the port it **fails loud** rather than silently adopting it (regtest `9739`, testnet `9738`, mainnet `9737`) — so a regtest run can never quietly attach to your mainnet engine.
- **v2 cooperative redeem is not RBF-bumpable.** Its fee is sealed into the pre-signed adaptor signature, and the transaction is broadcast non-replaceable to say so honestly (v2 fundings are too — the engine bumps both via CPFP). The fee commits at swap start at the live market rate for the coin's funding confirmation class, and a CPFP child bumps the redeem if the network gets busy — and that child is **deadline-aware**, stepping up as the timelock nears (bands at 6 h / 3 h / 90 min out) and re-checking every scheduler tick inside the final 90 minutes, so a stuck redeem still confirms. (The v2 single-key refund *is* bumpable; v1 redeem and refund are both bumpable, and redeem bumps are deadline-aware the same way. Refund fees never escalate — a matured timelock's refund is the sole possible spender, so there is no race to win.)
- **Fees never block a claim.** The flat operator fee ceiling (default 500 sat/vB) applies to *funding* only. A redeem or refund is bounded instead by the value it is claiming, so during a genuine fee spike near a timelock it can still pay the going rate and confirm rather than being capped and losing the leg.
- **The taker never locks second-in-the-dark.** In a v2 (Taproot) swap the taker builds and pre-signs its leg but broadcasts it only once both adaptor signatures are verified *and* the maker's leg is confirmed on-chain — so it can always claim the maker's leg before committing its own.
- **Liveness depends on relays.** If every board you use goes dark mid-swap, coordination stalls — but your timelock refund still protects the funds.
- **A dead machine no longer strands an in-flight swap.** `pactd` opportunistically backs up in-flight swap state to the same relays, encrypted **to its own identity key** — nobody else can read it. A machine restored from the seed alone can rediscover a swap the local database never saw. Restore is never automatic: `pactd` only ever detects and warns; adopting requires the explicit `restorefromrelay` RPC (or `pact-cli restore`), because two live machines driving the same swap on one seed could double-fund it. Only swaps started after this shipped are covered — an already-in-flight swap at upgrade time isn't retroactively rescuable.
- **One seed on several machines is safe — by partitioning, not election.** Each install holds a random derive scope (`machine.json`), injected as hardened levels into every counter-based swap derivation, so two machines on the same seed derive *different* secrets and swap ids at the same counter and can never collide. Another machine's swap shows up **followed**: read-only chain monitoring, never broadcast — it becomes driven here only through one explicit `takeover` confirm ("that machine is stopped"). Takeover never skips a swap: a v2 swap whose pinned cooperative-redeem payout this machine's wallet doesn't own is adopted **refund-only** (the driver refuses to complete to the foreign wallet and rides it to the timelock refund, which pays an address it owns), and an adopted swap with nothing locked yet broadcasts a fresh funding only when the chain positively proves the leg unfunded — anything inconclusive fails closed into a clean pre-funding abort. On restart, a machine reconciles every swap it drives against the chain before acting (once per process start, zero broadcasts), so swaps a same-seed machine settled in the meantime snap to their true terminal state instead of arming a refund against an already-spent leg. The limitation is coherence, not safety: N machines share **one** wallet balance (interchangeable drivers, not N× liquidity), so shared-wallet races like input-race errors or a stale standby balance can occur. Full design: `docs/MULTI_MACHINE_122.md`.
- **Every in-flight entry is cancellable before it costs you anything.** `abort` now covers a v1 record, a v2 (Taproot) record, and an unanswered pending take alike, always refusing once your own leg is funded (the timelock refund is the exit past that point). A v2 handshake stalled before either leg funds also self-cancels after 15 minutes with no action from you, and a board `take` older than that same window is dropped by the maker rather than served — so a counterparty that went quiet mid-handshake can never burn a live offer or strand an uncancelable record.
