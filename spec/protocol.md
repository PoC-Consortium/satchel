# Pact Swap Protocol v1 — HTLC atomic swaps Bitcoin PoCX ↔ BTC

Status: DRAFT (Phase 1). Protocol version string: `pact-htlc-v1`.

This document specifies the v1 atomic-swap protocol used by Pact: classic
hash-time-locked contracts (HTLCs) on two Bitcoin-derived UTXO chains, with
CHECKLOCKTIMEVERIFY (BIP65) refunds. It is written so that a third party can
implement an interoperable client without reading the Pact source code.

Keywords MUST / MUST NOT / SHOULD / MAY are RFC 2119.

---

## 1. Overview and roles

Two parties swap coins across two chains without ever trusting each other
with funds:

- **Alice** (the *initiator*) holds PoCX and wants BTC. She generates the
  secret preimage `s` and its hash `H = SHA256(s)`.
- **Bob** (the *participant*) holds BTC and wants PoCX.

Chain naming: **chain A** is the chain Alice locks funds on (PoCX), **chain
B** is the chain Bob locks funds on (BTC). The protocol is symmetric in the
assets: a BTC-holder-initiates swap is the same protocol with A=BTC, B=PoCX.

Flow:

1. Alice and Bob agree on terms (amounts, timelocks `T1`, `T2`, pubkeys)
   via the signed message handshake (§8).
2. Alice funds an HTLC on chain A: spendable by Bob with `s`, refundable to
   Alice at time `T1`.
3. Bob verifies the chain-A HTLC on-chain, then funds an HTLC on chain B
   with the **same `H`**: spendable by Alice with `s`, refundable to Bob at
   time `T2`, where `T2 < T1`.
4. Alice redeems the chain-B HTLC. Doing so publishes `s` in the witness.
5. Bob extracts `s` from Alice's redeem transaction and redeems the chain-A
   HTLC.

Either both legs complete, or — after the timelocks — both refund. Nobody
sends first.

v1 restrictions (by design — see docs/TRADING_ROADMAP.md):

- Fixed-size offers; the HTLC amount is exact; **no partial fills**.
- One HTLC per chain per swap.
- P2WSH only (no Taproot; adaptor-signature swaps are v2).

## 2. Notation and constants

| Symbol | Meaning |
|---|---|
| `s` | 32-byte secret preimage, generated by Alice (§4.3) |
| `H` | `SHA256(s)`, 32 bytes |
| `T1` | absolute refund time of the chain-A HTLC (Unix seconds) |
| `T2` | absolute refund time of the chain-B HTLC (Unix seconds), `T2 < T1` |
| `amount_A`, `amount_B` | HTLC amounts in base units (1 coin = 100 000 000 base units on both chains) |
| `P_a^A` | Alice's refund pubkey on chain A (compressed, 33 bytes) |
| `P_a^B` | Alice's redeem pubkey on chain B (compressed, 33 bytes) |
| `P_b^A` | Bob's redeem pubkey on chain A (compressed, 33 bytes) |
| `P_b^B` | Bob's refund pubkey on chain B (compressed, 33 bytes) |
| `hash160(x)` | `RIPEMD160(SHA256(x))` |
| `TaggedHash(tag, m)` | `SHA256(SHA256(tag) ‖ SHA256(tag) ‖ m)` with `tag` an ASCII string (BIP340 style) |

All hex in this document is lowercase. All amounts are integers in base
units; floating point MUST NOT appear in any message.

## 4. Keys and secrets

### 4.1 The Pact seed

Each party has a BIP39 seed owned by their Pact instance ("the Pact seed").
It holds only hot transit keys; swap proceeds sweep to the user's core
wallet. All per-swap material is derived from it via BIP32 so that the seed
plus the persisted handshake transcript recovers any swap (§11).

The BIP32 **purpose** for Pact is `7228'` ("PACT" on a phone keypad).

| Material | Path | Key type |
|---|---|---|
| Identity key | `m/7228'/0'/0'` | BIP340 x-only Schnorr key (message signing, §8.2; stable identity across swaps) |
| Swap key, chain *c*, swap index *i* | `m/7228'/1'/coin(c)'/i'` | compressed secp256k1, ECDSA (HTLC redeem/refund) |
| Preimage source, swap index *i* | `m/7228'/2'/i'` | private key bytes feed §4.3; never used as a signing key |

`coin(c)` constants (asset, not network — the same path is used on mainnet,
testnet and regtest; only address encoding differs):

| Asset | `coin(c)` |
|---|---|
| BTC | `0'` |
| PoCX | `20559'` (`0x504F`, ASCII "PO") |

`i` is the party's local, monotonically increasing swap index, starting at
0. Each party numbers their own swaps; the two parties' indices are
unrelated. A party uses **one swap key per chain per swap**: on the chain
where they lock funds it is their refund key; on the chain where they claim
it is their redeem key.

### 4.2 Key usage rules

- Swap keys MUST NOT be reused across swaps. Implementations MUST bump `i`
  for every new swap attempt, including aborted ones.
- The identity key MUST NOT be used in any HTLC.
- HTLC keys are compressed 33-byte secp256k1 pubkeys; signatures are ECDSA,
  DER-encoded, with `SIGHASH_ALL` (0x01).

### 4.3 Deterministic preimage

Alice (only) derives, for her swap index `i`:

```
k  = private key at m/7228'/2'/i'          (32 bytes, big-endian scalar)
s  = TaggedHash("pact/htlc/preimage/v1", k)
H  = SHA256(s)
```

`s` is exactly 32 bytes. Because `s` is derived from the seed, losing the
state database never loses the secret: the seed re-derives it (§11).

`s` MUST NOT be disclosed to anyone before Alice's redeem transaction is
broadcast (§9.4 governs *when* broadcasting it is safe).

### 4.4 Swap identifier

```
swap_id = hex( TaggedHash("pact/swapid/v1", H) [0..8) )    # 16 hex chars
```

Used in messages, CLI commands and logs. It leaks nothing about `s` and can
be shared freely.

## 5. The HTLC script

Both chains use the identical witness script (only the keys, hash and
locktime differ). It is the audited classic template (cf. Decred
`atomicswap`), with a preimage size guard:

```
OP_IF
    OP_SIZE 32 OP_EQUALVERIFY
    OP_SHA256 <H> OP_EQUALVERIFY
    OP_DUP OP_HASH160 <hash160(redeem_pubkey)>
OP_ELSE
    <T> OP_CHECKLOCKTIMEVERIFY OP_DROP
    OP_DUP OP_HASH160 <hash160(refund_pubkey)>
OP_ENDIF
OP_EQUALVERIFY
OP_CHECKSIG
```

Exact serialization (byte level):

```
63              OP_IF
  82            OP_SIZE
  01 20         push 1 byte: 0x20 (32)
  88            OP_EQUALVERIFY
  a8            OP_SHA256
  20 <H:32>     push 32 bytes: H
  88            OP_EQUALVERIFY
  76            OP_DUP
  a9            OP_HASH160
  14 <rpkh:20>  push 20 bytes: hash160(redeem_pubkey)
67              OP_ELSE
  <T as minimally-encoded CScriptNum push; 4-byte push for current epochs>
  b1            OP_CHECKLOCKTIMEVERIFY
  75            OP_DROP
  76            OP_DUP
  a9            OP_HASH160
  14 <fpkh:20>  push 20 bytes: hash160(refund_pubkey)
68              OP_ENDIF
88              OP_EQUALVERIFY
ac              OP_CHECKSIG
```

Instantiation per chain:

| | chain A (Alice locks) | chain B (Bob locks) |
|---|---|---|
| redeem_pubkey | `P_b^A` (Bob) | `P_a^B` (Alice) |
| refund_pubkey | `P_a^A` (Alice) | `P_b^B` (Bob) |
| `T` | `T1` | `T2` |

Rules:

- `T` MUST be a Unix timestamp `>= 500_000_000` (time-based locktime,
  BIP65). Height-based locktimes MUST NOT be used in v1 — the two chains
  have different block cadences; wall-clock time is the common unit.
- `T` MUST be minimally encoded (CScriptNum). For all realistic times
  (< 2³¹) this is a 4-byte little-endian push.
- The HTLC output is **P2WSH**: `scriptPubKey = OP_0 <SHA256(witnessScript)>`
  (`00 20 <32 bytes>`). P2SH and bare scripts MUST NOT be used.
- Address encoding: bech32 (witness v0) with the HRP from §3 — `pocx…` /
  `rpocx…` on PoCX, `bc…` / `bcrt…` on Bitcoin.

**Verification rule (critical):** a party MUST reconstruct the counterparty
HTLC witness script locally from the agreed parameters, compute the P2WSH
scriptPubKey, and compare it byte-for-byte against the output found
on-chain. A script or address received in a message MUST NOT be trusted.

## 6. Transactions

All transactions are version 2. Standardness rules of stock Bitcoin Core
apply on both chains.

### 6.1 Funding transaction

Any transaction that creates an output of exactly `amount` to the P2WSH
scriptPubKey of §5. Typically a normal send from the party's core wallet;
its other inputs/outputs (change) are unconstrained. The funder communicates
`(txid, vout)` in a `funded` message (§8.4). The counterparty MUST verify
on-chain, via their own backend:

1. the output's scriptPubKey equals the locally reconstructed P2WSH;
2. the output's value equals the agreed amount exactly;
3. the required confirmation depth (§7.3).

### 6.2 Redeem transaction

Spends the counterparty's HTLC output via the hash branch.

| Field | Value |
|---|---|
| version | 2 |
| input | the HTLC outpoint; `nSequence = 0xfffffffd` (RBF-signaling) |
| witness | `[ <DER sig ‖ 0x01>, <33-byte redeem pubkey>, <s (32 bytes)>, <0x01>, <witnessScript> ]` |
| scriptSig | empty |
| nLockTime | 0 |
| output(s) | one output paying `amount − fee` to the redeemer's destination |

The `0x01` witness element selects the `OP_IF` branch. Signatures commit
with `SIGHASH_ALL` under BIP143.

The redeem destination SHOULD be a fresh address of the redeemer's **core
wallet** (not a Pact-seed address): proceeds sweep out of the hot seed in
the same transaction that claims them.

### 6.3 Refund transaction

Spends one's own HTLC output via the timeout branch.

| Field | Value |
|---|---|
| version | 2 |
| input | the HTLC outpoint; `nSequence = 0xfffffffd` |
| witness | `[ <DER sig ‖ 0x01>, <33-byte refund pubkey>, <> (empty), <witnessScript> ]` |
| scriptSig | empty |
| nLockTime | `T` (the HTLC's locktime, exactly) |
| output(s) | one output paying `amount − fee` to the refunder's destination |

The empty witness element selects the `OP_ELSE` branch. The transaction is
only valid once the chain's median-time-past (BIP113) reaches `T`; see
§7.2 for the implied delay.

Implementations MUST construct and sign the refund transaction **at funding
time**, persist it, and schedule its broadcast at `T` independent of any UI
being open. Refund broadcasting MUST be retried until confirmed.

### 6.4 Fees

Redeem/refund fees come out of the HTLC value (`output = amount − fee`).
The spender picks the feerate. The output MUST remain above the dust limit
(546 base units conservative bound); offers SHOULD be sized so that fees are
a negligible fraction of the amount. Both redeem and refund signal RBF and
MUST be fee-bumped if unconfirmed while their deadline (§7) approaches.

## 7. Timelocks and confirmation policy

### 7.1 Structural rules (all networks, MUST)

- `T2 < T1`. Alice (secret holder) gets the *longer* refund timeout.
  `δ = T1 − T2` is Bob's guaranteed window: even if Alice reveals `s` at
  the last valid moment before `T2`, Bob has at least `δ` to redeem on
  chain A before Alice could refund there.
- Both HTLCs use absolute time-based CLTV (§5).
- Timelocks are fixed at handshake time and MUST NOT be renegotiated after
  any funding transaction is broadcast.

### 7.2 Median-time-past lag

BIP113 evaluates locktimes against the median of the last 11 block
timestamps, which trails wall-clock time — typically by ~1 hour on Bitcoin
(10-min blocks) and ~12 minutes on PoCX (2-min blocks), more if hashrate or
capacity fluctuates. Consequences:

- A refund broadcast at wall-clock `T` will be rejected until MTP ≥ `T`.
  Schedulers MUST keep retrying; they MUST NOT treat early rejection as
  failure.
- Safety margins in §7.4 already account for adversarial timestamp skew
  (up to ~2 h on the BTC side).

### 7.3 Network profile: mainnet/testnet (normative defaults)

With `t0` = the time Alice sends `init`:

| Parameter | Value | Constraint (MUST) |
|---|---|---|
| `T2` | `t0 + 6 h` (default) | `T2 ≥ t0 + 3 h` |
| `T1` | `t0 + 12 h` (default) | `T1 − T2 ≥ 4 h`, `T1 ≤ t0 + 48 h` |
| `N_A` (confs of chain-A HTLC before Bob funds) | 10 on PoCX (~20 min) | ≥ 6 |
| `N_B` (confs of chain-B HTLC before Alice redeems) | 3 on BTC (~30 min) | ≥ 1 |

"Hours, not days": the 48 h cap bounds the free-option problem (a stalling
counterparty holds at most a 48 h option; defaults hold a 6 h one).

If the asset roles are reversed (initiator locks BTC), swap the `N` values
with the chains: the chain with 120 s blocks always uses the higher count.

### 7.4 Action deadlines (MUST)

These rules prevent the known race losses. Times compare the party's local
clock (NTP-synced) against the agreed timestamps.

- **Bob funds only early enough:** Bob MUST NOT broadcast his funding
  transaction after `T2 − 3 h`. (Funding later shrinks Alice's redeem
  window to nothing; she will abort and both will refund.)
- **Alice's reveal deadline:** Alice MUST NOT broadcast her redeem
  transaction after `T2 − 2 h`, and MUST NOT broadcast it at all unless
  the chain-B HTLC already has `N_B` confirmations. If her redeem is
  unconfirmed as `T2` approaches she MUST fee-bump aggressively. Rationale:
  a redeem that lingers in the mempool past `T2` reveals `s` while Bob can
  already refund chain B — Bob could then take *both* legs.
- **Alice aborts cleanly:** if the chain-B HTLC has not reached `N_B`
  confirmations by `T2 − 3 h`, Alice MUST give up on redeeming, never
  reveal `s`, and wait for her `T1` refund.
- **Bob redeems promptly:** once `s` is known, Bob SHOULD broadcast his
  chain-A redeem immediately and MUST do so before `T1 − 1 h`,
  fee-bumping as needed.

### 7.5 Network profile: regtest

Structural rules (§7.1) still apply; all durations and confirmation counts
are free (tests typically use minutes and `N = 1`). On PoCX regtest, block
forging enforces a minimum inter-block delay unless `setmocktime` is active;
test harnesses drive time explicitly with `setmocktime` on both nodes, which
also makes MTP-based refund tests deterministic.

## 8. Message handshake

### 8.1 Transport and encoding

Messages are transport-agnostic signed JSON objects. In Phase 1 they are
exchanged manually (files / copy-paste); later transports (Corkboard relay,
Nostr) carry the same bytes. A swap uses exactly this sequence:

```
Alice                                   Bob
  | -- init ---------------------------> |
  | <-------------------------- accept --|
  |        (Alice funds chain A)         |
  | -- funded(A) ----------------------> |
  |        (Bob verifies, funds chain B) |
  | <----------------------- funded(B) --|
  |        (Alice redeems B, reveals s)  |
  | -- redeemed (courtesy) ------------> |
  |        (Bob redeems A)               |
```

Either side may send `abort` before their own funding tx is broadcast;
after funding, `abort` is advisory only — safety comes from timelocks, not
messages.

**Canonical encoding** (for signing): UTF-8 JSON with object keys sorted
lexicographically (byte order), no insignificant whitespace, integers in
decimal without leading zeros, binary values as lowercase hex strings.
Floats MUST NOT appear.

### 8.2 Envelope and signature

Every message is:

```json
{
  "v": 1,
  "type": "init|accept|funded|redeemed|abort",
  "swap_id": "<16 hex chars>",
  "from": "<32-byte x-only identity pubkey, hex>",
  "body": { ... },
  "sig": "<64-byte BIP340 Schnorr signature, hex>"
}
```

`sig` is computed over `TaggedHash("pact/msg/v1", C)` where `C` is the
canonical encoding (§8.1) of the envelope **without** the `sig` field,
signed with the sender's identity key (§4.1). Receivers MUST verify `sig`
against the `from` key, and that `from` matches the identity pinned in the
first message received from that party for this `swap_id`.

### 8.3 `init` (Alice → Bob)

```json
"body": {
  "protocol": "pact-htlc-v1",
  "chain_a": {"asset": "btcx", "network": "regtest"},
  "chain_b": {"asset": "btc",  "network": "regtest"},
  "amount_a": 5000000000,
  "amount_b": 100000,
  "hash_h": "<32 bytes hex>",
  "t1": 1781000000,
  "t2": 1780978400,
  "n_a": 10,
  "n_b": 3,
  "alice_refund_pubkey_a": "<33 bytes hex>",
  "alice_redeem_pubkey_b": "<33 bytes hex>"
}
```

`swap_id` MUST equal §4.4 applied to `hash_h`. Bob MUST validate: protocol
string, chains/network match what he expects, amounts match the agreed
offer, §7 constraints on `t1`/`t2` against his own clock, and pubkeys are
valid curve points. If anything fails he replies `abort`.

### 8.4 `accept` (Bob → Alice)

```json
"body": {
  "bob_redeem_pubkey_a": "<33 bytes hex>",
  "bob_refund_pubkey_b": "<33 bytes hex>"
}
```

After `accept`, both sides can deterministically construct both witness
scripts. Each side MUST persist the full transcript before any funding
(§11).

### 8.5 `funded` (funder → counterparty)

```json
"body": { "chain": "a", "txid": "<32 bytes hex, big-endian display order>", "vout": 0 }
```

Sent after broadcasting the funding transaction (Alice with `chain":"a"`,
Bob with `"chain":"b"`). The receiver performs the §6.1 verification
against the chain — the message is a pointer, not a proof.

### 8.6 `redeemed` (Alice → Bob, courtesy)

```json
"body": { "preimage": "<32 bytes hex>" }
```

Bob MUST NOT rely on receiving this: he MUST independently watch the
chain-B HTLC outpoint and extract `s` from the spending transaction's
witness (third witness element, validated by `SHA256(s) == H`).

### 8.7 `abort` (either direction)

```json
"body": { "reason": "<human-readable string>" }
```

### 8.8 Nostr transport (optional, non-normative)

The envelopes above are transport-agnostic (§8.1). Besides manual transport
and the HTTP Corkboard, an implementation MAY carry them over **Nostr**
relays. This binding changes nothing about the swap: it transports the same
signed envelopes and the same `PACTSEALED1` relay blobs (§10). See
[`../docs/NOSTR_TRANSPORT.md`](../docs/NOSTR_TRANSPORT.md) for the full design;
the wire mapping is:

- **Identity.** A party's Nostr key IS its Pact identity key — both are the
  same BIP340 x-only secp256k1 key, so `npub` == the `from` pubkey of §8.2.
- **Offer** → an **addressable** event, `kind 31510`, `d` = `swap_id`, with
  the signed `offer` envelope as `content` (so §8.2 verification still
  applies) and a NIP-40 `["expiration", created+ttl_secs]` tag. Discovery
  subscribes by kind; pair/network filtering is client-side. Revocation is a
  NIP-09 deletion of the coordinate `31510:<pubkey>:<swap_id>`.
- **Relay messages** (`take`/`init`/`accept`/`funded`/`redeemed`/`abort`) →
  a **gift wrap**, `kind 1059`, signed by a fresh one-time key and
  `#p`-tagged to the recipient, with the §10 sealed blob as `content`. The
  ephemeral author hides the sender from relays; the seal hides the contents
  from everyone. The recipient subscribes to `kind 1059` `#p`=self and opens
  each blob exactly as for the HTTP relay.

A relay that withholds or delays events affects liveness only — never
safety, which rests on timelocks (§10), identical to the corkboard relay.

## 9. Procedures

### 9.1 Alice: fund chain A

Preconditions: `accept` received and validated; transcript persisted.

1. Construct the chain-A witness script (§5) and P2WSH address.
2. Fund it with exactly `amount_a` from the core wallet.
3. Construct + sign the `T1` refund transaction now; persist it; schedule
   broadcast at `T1` (§6.3).
4. Send `funded(a)`.

### 9.2 Bob: verify and fund chain B

1. Reconstruct the chain-A script locally; locate `(txid_a, vout_a)`;
   verify scriptPubKey, exact amount, and `N_A` confirmations.
2. Check the clock: refuse to fund after `T2 − 3 h` (§7.4).
3. Construct the chain-B witness script, fund with exactly `amount_b`.
4. Construct + sign + persist the `T2` refund; schedule it.
5. Send `funded(b)`.

### 9.3 Alice: redeem chain B (reveals `s`)

1. Verify the chain-B HTLC on-chain (script bytes, amount, `N_B` confs).
2. Enforce §7.4: no redeem after `T2 − 2 h`; abort to refund path instead.
3. Broadcast the redeem (§6.2) paying her core wallet; fee-bump until
   confirmed; cancel her scheduled `T1` refund only once the redeem is
   confirmed `N_B` deep.

### 9.4 Bob: extract `s`, redeem chain A

1. On seeing the chain-B HTLC spent, extract `s`, check `SHA256(s) = H`.
2. Broadcast the chain-A redeem paying his core wallet; before `T1 − 1 h`;
   fee-bump until confirmed; then cancel his scheduled `T2` refund.

### 9.5 Refund paths

- Alice: if she never reveals `s`, her `T1` refund returns `amount_a`.
- Bob: if Alice never redeems chain B, his `T2` refund returns `amount_b`.
- A refund MUST stay scheduled until the corresponding redeem is confirmed;
  "redeem broadcast" is not "redeem confirmed".

## 10. Adversarial assumptions and safety arguments

**Chain data may lie (Electrum servers).** Safety MUST NOT depend on honest
chain-data servers:

- Servers never see keys or `s`; they cannot steal.
- All scripts/amounts are verified against locally reconstructed bytes, so
  a server cannot substitute a bad HTLC.
- A server *can* withhold or delay information. Refund scheduling is purely
  local (clock-driven) and unaffected. The one liveness-critical read is
  Bob learning `s` before `T1`: the `δ = T1 − T2 ≥ 4 h` window plus the
  requirement below bound this risk.
- Each party SHOULD monitor through at least two independent backends (own
  node, or ≥ 2 unrelated Electrum servers) during the life of a swap, and
  MUST treat backend silence near a deadline as an emergency (switch
  servers, raise alarms) rather than as "nothing happened".

**Free option.** Between funding and `T2`, Alice holds an option: complete
if the price moved her way, refund otherwise. v1 mitigations: short
timelocks (§7.3), small fixed offer sizes, and maker-funds-first ordering.
Bob prices this in; the protocol cannot remove it in v1.

**Reveal-too-late race.** Covered by the §7.4 deadlines: `s` must never be
in any mempool at a time when Bob's refund of chain B could confirm.

**Reorgs.** Confirmation requirements `N_A`, `N_B` (§7.3) bound reorg risk;
PoCX's higher count reflects its younger, smaller-capacity chain. After a
reorg that unconfirms a counterparty HTLC, parties MUST re-verify from
scratch and re-apply all deadlines.

**Clock.** Both parties need NTP-grade clocks. A party whose clock is wrong
by more than the §7.4 margins can lose funds; implementations SHOULD check
their clock against both chains' tip timestamps and refuse to operate on
gross divergence.

## 11. Persistence and recovery

Implementations MUST persist durably, before broadcasting any funding
transaction: the full message transcript (all envelopes, both directions),
their own swap index `i`, the signed refund transaction, and the swap state.

Recovery story:

- **Seed only:** re-derives identity, all swap keys, and all preimages
  (§4.3) — `s` is never lost. It does not reconstruct counterparty
  parameters, so in-flight swap *outputs* cannot be located from the seed
  alone.
- **Seed + transcript:** full recovery of any swap at any state — scripts
  reconstruct from transcript parameters, refunds re-sign from derived
  keys, chain state re-syncs by scanning for the known outpoints.

Therefore the transcript is part of the safety state, not a cache. The
SQLite state file is the convenient form of it; backups SHOULD include it,
and implementations SHOULD also write each message to an append-only file
as it is sent/received.

## 12. Versioning

The protocol string `pact-htlc-v1` covers exactly: the script of §5, the
transactions of §6, the rules of §7, and the messages of §8. Any
incompatible change bumps to `pact-htlc-v2` (planned: Taproot
adaptor-signature swaps, MuSig2). A party receiving an `init` with an
unknown protocol string MUST reply `abort`.
